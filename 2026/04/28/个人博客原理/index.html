1.个人博客原理详解

基于 Hexo + GitHub Pages + Typora + PicGo 的零基础

工具 作用
Hexo 基于 Node.js 的静态博客生成器，把 Markdown 文章编译成可部署的网页
GitHub Pages 免费静态网站托管服务，用来存放 Hexo 生成的博客文件，提供公网访问
Typora 轻量 Markdown 编辑器，用来写博客文章，实时预览、所见即所得
PicGo 图床上传工具，一键把文章里的本地图片上传到图床，生成在线链接

我们一步步拆解 GitHub Pages 托管 Hexo 博客的原理，把它拆成「你电脑里的 Hexo」和「GitHub 服务器上的 Pages」两部分来看，就很容易懂了👇

一、你电脑里的 Hexo，是怎么工作的？

Hexo 本质上是一个静态网站生成器，它的工作流程是这样的：

  1. 写文章：你用 Markdown 写的博客文章，都存在 source/_posts 文件夹里。

  2. 生成静态文件

    ：执行

    命令时，Hexo 会做两件事：

    • 把你的 Markdown 文章，转换成浏览器能直接打开的 .html 文件
    • 把主题的样式、脚本、图片等资源，打包成完整的静态文件，存放在 public 文件夹里

  3. 本地预览：执行 hexo s 命令，Hexo 会在你电脑上启动一个本地服务器，把 public 文件夹里的文件跑起来，所以你能在 http://localhost:4000 看到效果。

二、GitHub Pages，是怎么把你的博客放到网上的？

GitHub Pages 是 GitHub 提供的免费静态网站托管服务，它的核心逻辑是：

只要你把网站的静态文件（index.html、CSS、JS 等）推送到仓库，GitHub 就会帮你托管，让全世界都能通过你的仓库名 .github.io 域名访问。

结合 Hexo，完整的部署流程是这样的：

  1. 推文件：执行 hexo d 命令，会把 public 文件夹里的所有静态文件，推送到你的 goglefeng.github.io 仓库的 master 分支上。
  2. 构建网站：你在仓库设置里启用了 GitHub Pages，并指定了 master 分支，GitHub 就会自动读取这个分支里的文件，构建你的网站。
  3. 提供访问：构建完成后，GitHub 会给你的仓库分配一个 https://goglefeng.github.io 地址，任何人访问这个地址，都会被指向你仓库里的 index.html，也就是你的博客首页。

三、为什么之前会 404？

之前的 404 问题，是因为 GitHub Pages 没读到你的文件，主要有两个原因：

  1. 仓库是私有的：免费版 GitHub 私有仓库无法启用 Pages，GitHub 不会为它构建网站。
  2. Pages 没配置好：仓库公开后，没有在设置里指定 Pages 的构建分支，GitHub 不知道从哪里读取文件。

现在你把仓库改成了公开，也设置了从 master 分支构建，GitHub 就能正常读取你推送的静态文件，构建并托管你的博客了。

💡 总结一下

  • Hexo：负责把你的文章和主题，转换成能直接被浏览器打开的静态文件。
  • GitHub Pages：负责把这些静态文件托管到服务器上，并提供公开的访问地址。
  • 部署命令：就是把 Hexo 生成的文件，上传到 GitHub，让 Pages 能读到它们。

简单说，就是你写的文章，经过 Hexo 加工，变成了网页文件，再被 GitHub Pages 放到了网上，全世界都能访问啦。
